# MacSweep — Repo Map (Audit 00)

Date: 2026-07-02
Auditor: Claude Code (repo-map session; evidence-first, no code changes)
Repo: `github.com/VincentShipsIt/macsweep.dev`, branch snapshot at `2505479` (`release: bump version to 1.0.7`)

Scope note: the brief said "monorepo or group of monorepos." **This is neither** — it is a single-product repository: one macOS application with a companion CLI, built from one shared Swift codebase. There are no workspaces, no `package.json`, no JS/TS toolchain, no multiple deployable services. Everything below reflects that reality.

---

## 1. System overview

MacSweep is a native macOS system cleaner (CleanMyMac-style) with three delivery surfaces built from one Swift codebase:

1. **GUI app** (`MacSweep.app`) — SwiftUI + AppKit, macOS 26 "Tahoe" minimum, with a menu-bar widget, dashboard, and ~20 feature screens (`MacSweep/Sources/App/MacSweepApp.swift`, `MacSweep/Sources/Features/`).
2. **CLI** (`macsweep`) — a hand-rolled command parser with ~34 subcommands and `text`/`json` output (`MacSweep/Sources/CLI/main.swift`, `MacSweep/Sources/CLIKit/CLICommand.swift:10-43`).
3. **Headless service layer** — the shared programmatic API both surfaces call (`MacSweep/Sources/Core/Headless/HeadlessService.swift`), with Codable result types for JSON serialization (`HeadlessModels.swift`, 1,084 lines).

Core domain: a 19-module scan/clean engine (`ScanEngine.swift:78-109`) behind a default-deny safety layer (`SafetyChecker.swift`), plus adjacent utilities: malware persistence scanner, Homebrew updater, login-items manager, secure shredder, disk visualizer ("Space Lens"), system monitoring, and an AI assistant that plans cleanups via local `claude`/`codex` CLIs or the Anthropic API.

Key numbers:

- 184 tracked files; 93 Swift source files (~34,200 LOC) + 21 test files (~3,100 LOC, 264 `@Test` functions).
- **Zero external package dependencies** — `MacSweep/Package.swift` declares no `dependencies:` at all. Everything (TOML codec, keychain, arg parsing) is hand-rolled.
- License: MIT (`LICENSE`). README carries a "work in progress" warning banner (`README.md:3-6`).
- Version: 1.0.7 (`MacSweep/project.yml:11`, git tag `v1.0.7`).

### Build targets — two parallel build systems, deliberate split

| Target | System | Product | Evidence |
|---|---|---|---|
| `MacSweepCore` | SPM library + XcodeGen framework | shared core | `Package.swift:31-34`, `project.yml:50-59` |
| `MacSweepCLIKit` | SPM | CLI logic lib | `Package.swift:35-39` |
| `MacSweepCLI` | SPM executable + XcodeGen tool | `macsweep` binary | `Package.swift:40-44`, `project.yml:61-74` |
| `MacSweep` (app) | XcodeGen/xcodeproj only | `MacSweep.app` | `project.yml:18-48` |
| `MacSweepTests` | SPM test target | swift-testing suite | `Package.swift:45-49` |

The GUI app is **not** buildable via SPM (Features/App excluded); the CLI **is** buildable via SPM and that is exactly what CI and the Homebrew formula build (`.github/workflows/ci.yml:18-45`). The generated `.xcodeproj` is committed and must be regenerated with `xcodegen` on version bumps (`scripts/release.sh:87-98`).

Version consistency is enforced across three files by `scripts/release.sh check`: `project.yml` `MARKETING_VERSION`, `Sources/Core/MacSweepVersion.swift`, and the generated pbxproj (`scripts/release.sh:7-13,57-81`).

## 3. Runtime / deployment map

- **Runtime**: local macOS only. No servers, no cloud runtime, no database. Minimum OS macOS 26.0 (`Package.swift:24`, `project.yml:5`).
- **Language/toolchain**: Swift tools-version 6.2, but sources pinned to **Swift 5 language mode** — explicitly documented as "not audited for Swift 6 strict concurrency" (`Package.swift:51-56`). `project.yml` sets `SWIFT_VERSION: "5.9"` per target.
- **Distribution** (all via Homebrew, tap repo `VincentShipsIt/homebrew-tap`):
  - CLI: build-from-source formula `Formula/macsweep.rb`.
  - GUI: prebuilt-zip cask `Casks/macsweep.rb` that `depends_on formula` and `depends_on macos: ">= :tahoe"` (`.github/workflows/update-homebrew.yml:106-122`).
  - GitHub Releases carry `macsweep-<tag>-macos.zip` + SHA256 files (`.github/workflows/release.yml:89-100`).
- **Signing/notarization**: **none**. Builds are ad-hoc signed (`CODE_SIGN_IDENTITY="-"`, empty `DEVELOPMENT_TEAM`) in both CI and release (`ci.yml:18-45`, `release.yml:44-51`); `scripts/release.sh:21` states "No code signing / notarization yet". Hardened runtime is enabled in project.yml but is moot without real signing.
- **Sandbox**: disabled by design (`com.apple.security.app-sandbox: false` in `MacSweep.entitlements:5-6`) — required for full-disk cleanup; also `automation.apple-events: true`.
- **Background execution**: no launchd daemon, no BGTask. A `@MainActor` in-app `Timer` schedules a recurring scan (default weekly, interval CLI-configurable), persisting next-run date in a shared `UserDefaults` suite so GUI and CLI processes coordinate (`Sources/Background/ScanScheduler.swift:1-50`, `Sources/Core/Services/SchedulerConfig.swift`). Scans therefore only run while the app is running.
- **Notifications**: `UNUserNotificationCenter` for scan-complete alerts (`Sources/Background/NotificationManager.swift`).
- **Self-update**: `macsweep self-update` delegates to `brew upgrade` (reports the command by default; runs it with `--apply`) (`HeadlessService.swift:839-847`, `CLIExecutor.swift:576-579`).

### External binaries shelled out to (evidence: repo-wide grep of `Process()`/`executableURL`)

System tools: `networksetup`, `periodic`, `sfltool`, `mdutil`, `diskutil`, `vm_stat`, `osascript`, `mkfile`, `mdls`, `dscacheutil`, `atsutil`, `ps`, `purge`, `lsregister`, `system_profiler`, `crontab`, `codesign` (via malware scanner). Third-party (probed, optional): `docker` (3 candidate paths incl. Apple Silicon Homebrew, `DockerModule.swift:8-13`), `brew`, `claude`, `codex` (resolved from `PATH`).

## 4. Data / auth / integration map

**No database, no user accounts, no auth provider, no queues, no server-side jobs.** State lives in:

| Store | What | Evidence |
|---|---|---|
| `UserDefaults` suite `dev.macsweep` | scan interval, next-scan date, last-scan summary (JSON blob) | `SchedulerConfig.swift:15-51`, `LastScanStore.swift:13-21` |
| `@AppStorage` | onboarding-complete flag | `MacSweepApp.swift:7` |
| macOS Keychain (generic password, service `dev.macsweep`, account `ai-api-key`) | single Anthropic API key | `AIKeychainService.swift:9-51` |
| TOML files under an assistant config root (`assistant/providers.toml`, `watchlists/watchlists.toml`, bootstrapped with defaults) | AI provider config + user watchlist rules; hand-rolled TOML codec | `AssistantConfigRepository.swift:1-60`, `AssistantTOMLCodec.swift` |

### AI integrations (the only network integrations in the product)

Two mechanisms:

1. **Local agent CLIs** — assistant and cache analyzer prefer shelling out to `claude -p` then `codex exec` found on `PATH` (`CacheAnalyzer.swift:185-223`, `AssistantConversationService.swift:178-197`; provider kinds `codex|claude|openai` in `AssistantModels.swift:3-6`).
2. **Direct Anthropic API** (`https://api.anthropic.com/v1/messages`, key from keychain) — **four separate hand-rolled call sites with three different hardcoded models**:
   - `CacheAnalyzer.swift:250-266` — `claude-haiku-4-5`
   - `MalwareScannerService.swift:521-531` — `claude-sonnet-4-5`
   - `HomebrewService.swift:251-258` — `claude-sonnet-4-5`
   - `LoginItemsService.swift:251-259` — `claude-opus-4-5` (this one lives in the **Features/UI layer**)

Data sent to AI: directory listings/cache paths, plist contents of launch agents, browser-extension metadata, login-item metadata, Homebrew package lists. The malware scanner explicitly hardens against prompt injection from scanned content — an AI "clean" verdict cannot override a deterministic suspicious signal (`MalwareScannerService.swift:484-498`).

### Safety architecture (the load-bearing subsystem)

- Default-deny allowlist validation for automated cleanup; unrecognized paths are refused (`SafetyChecker.swift:3-18`).
- Separate blocklist mode for explicit user actions (shred, Space Lens trash) with symlink and parent-symlink resolution defenses (`SafetyChecker.swift:56-118`).
- `DeletionGuard`: 10 GB hard cap per operation, 1 GB confirmation threshold, dry-run default (`SafetyChecker.swift:622-645`).
- Disposal policy split: recoverable (Trash) vs permanent, centralized in `CleanupFileRemover.swift`.
- Engine-level enforcement: every scan result and every cleanup item re-validated inside `ScanEngine` (actor), which also captures per-module failures as `PartialScanResult` instead of swallowing them (`ScanEngine.swift:152-222,242-285`).
- Shredder documents APFS/SSD limits honestly — overwrite passes cannot guarantee erasure; FileVault is named as the real guarantee (`SecureDelete.swift:4-44`).

## 5. CI / test / tooling map

### GitHub Actions (4 workflows)

| Workflow | Trigger | Runner | Does |
|---|---|---|---|
| `ci.yml` | push, PR, dispatch | `macos-26` | `swift build -c release --product macsweep`, `xcodebuild` app (ad-hoc sign), `./scripts/test.sh` |
| `release.yml` | tag `v*` | `macos-26` | rebuild CLI + app, zip via `ditto`, compute/validate SHA256s, publish GitHub Release, then directly invoke update-homebrew (reusable-workflow call, because a `GITHUB_TOKEN`-created release doesn't cascade-trigger events) |
| `update-homebrew.yml` | `workflow_call` + manual dispatch | `ubuntu-latest` | sed-edit formula + regenerate cask in `VincentShipsIt/homebrew-tap`, push as github-actions bot; needs `TAP_GITHUB_TOKEN` PAT |
| `secret-scan.yml` | push, PR, dispatch | `ubuntu-latest` | gitleaks 8.30.1, SHA256-verified binary, full-history scan (`fetch-depth: 0`), fails on any leak |

Release runbook is documented as a slash command (`.claude/commands/release.md`): preflight → `release.sh check` → annotated tag on `origin/master` → push tag → watch runs.

### Testing

- Framework: **swift-testing** (`import Testing`) in all 21 files; 264 `@Test` functions; run via `swift test` wrapped by `scripts/test.sh`.
- `scripts/test.sh` is unusually defensive: detects Xcode vs CommandLineTools-only hosts (CLT needs `--disable-xctest` + explicit framework/rpath flags) and guards against silent-pass (exit 0 with zero executed tests → forced failure) (`scripts/test.sh:18-75`; rationale documented in `Package.swift:4-19`).
- Coverage shape: safety layer, engine, CLI parser, malware pipeline, headless serialization, individual modules are tested. **Zero tests** for: all of `Features/` (136 view structs), `App/`, `Background/` (scheduler/notifications), `CLIExecutor` (1,158 ln — only the parser is tested), `AssistantConversationService`, `HomebrewService`, `SystemMonitor`, `MaintenanceActions`, `SecureDelete`.
- No UI tests, no snapshot tests in CI (the screenshot harness `scripts/render-screenshots.sh` exists but is not wired into any workflow).

### Developer tooling

- `xcodegen` required for version bumps (regenerates committed xcodeproj).
- No linter/formatter config anywhere (no SwiftLint/SwiftFormat file), no pre-commit hooks, no Dependabot/Renovate (nothing to update — zero deps — except GitHub Actions).
- `.agents/` doc system (PRD, roadmap, architecture, tasks) is present but drifting — see risks.

## 6. Main architectural risks noticed during mapping

Ordered by expected impact. These are observations from mapping, not a full audit.

1. **Unsigned, un-notarized distribution.** Ad-hoc signing everywhere (`ci.yml`, `release.yml`, `project.yml:13-16`); acknowledged in `scripts/release.sh:21`. Gatekeeper friction for every GUI user, and a trust ceiling for a tool that requests Full Disk Access and deletes files. Biggest single product risk.

2. **AI client duplication across layers.** Four hand-rolled Anthropic `URLRequest` call sites, three different hardcoded model IDs, inconsistent placement (one in the UI layer: `Features/LoginItems/LoginItemsService.swift:251`). Model deprecation or API change requires four synchronized edits. Also `Sources/Services/AIAnalysisService.swift` sits in an orphaned top-level dir that belongs in `Core/` or `Features/` (it is GUI-only: `@MainActor ObservableObject`).

3. **Swift 6 concurrency debt is explicit.** Tools 6.2 with `swiftLanguageModes: [.v5]` and a comment saying sources are unaudited for strict concurrency (`Package.swift:51-56`). Meanwhile the codebase mixes actors (`ScanEngine`, `AssistantConfigRepository`), `@MainActor` singletons, `@unchecked Sendable` (`SchedulerConfig.swift:15`), and Combine. The longer Features grows (34K LOC), the more expensive the migration.

4. **Business logic embedded in SwiftUI views.** Views shell out to system binaries directly (`OptimizationView.swift:239-240` runs `purge`; `CPUDetailView.swift:238` kills processes; `LoginItemsService` in Features calls the Anthropic API). This bypasses the Core/Headless layering that the CLI relies on, and is exactly the code with zero test coverage.

5. **Background scheduling only works while the app runs.** Timer-in-process (`ScanScheduler.swift:3-11` documents this consciously) — no launchd agent. A "weekly scan" silently never fires for users who quit the app. Cross-process coordination via shared `UserDefaults` suite is fragile if the GUI and CLI race.

6. **Committed-generated-file drift risk.** `MacSweep.xcodeproj` is generated from `project.yml` but committed; nothing in CI verifies `xcodegen` output matches the committed pbxproj (only `release.sh check` compares MARKETING_VERSION). A manual pbxproj edit or forgotten regen diverges silently. Package.resolved is gitignored (harmless today with zero deps, but wrong reflex if deps arrive).

7. **Docs/version bookkeeping drift (found ≥3 instances — a pattern).**
   - `CHANGELOG.md` missing entries for 1.0.3, 1.0.4, and current 1.0.7 (tags exist for all; changelog stops at 1.0.6).
   - `.agents/SYSTEM/ARCHITECTURE.md` documents an obsolete `ScanModule` signature (`clean(_ items:) -> Int` vs actual `clean(items:dryRun:) -> CleanupResult`, `ScanModule.swift:14`) and omits `CLIKit/`, `CLI/`, `Background/`, `Assistant/`, `Maintenance/` from its directory tree.
   - `.gitignore:36-38` ignores `.agents/TASKS/*.md` yet 16 TASK files are tracked (committed before the rule) — ambiguous source-of-truth; task statuses: 5 Done, 9 Backlog, 2 "AI Review".
   - README's Swift badge says 6.2+ while targets build in 5.9/v5 language mode — technically both true, but confusing.

8. **Supply-chain pinning inconsistency in CI.** `release.yml`/`update-homebrew.yml` pin actions to commit SHAs; `ci.yml` uses floating `actions/checkout@v4`. Neither macOS workflow pins an Xcode version — builds ride whatever `macos-26` image ships, so toolchain changes arrive silently (carried over from the CI reader's findings).

9. **`bash -c` shell helper in malware scanner.** `MalwareScannerService.swift:823-827` runs string-composed commands through `/bin/bash -c`. Current two call sites are static strings (`:59`, `:144`) and a safe argv-based `run()` exists right below it (`:844`) — but the footgun remains available in the most security-sensitive file.

10. **Giant files accumulating.** `DevToolsModule.swift` 1,344 ln, `CLIExecutor.swift` 1,158 ln, `DashboardView.swift` 1,169 ln, `HeadlessModels.swift` 1,084 ln. Not yet pathological, but these four are where merge pain and review blindness will start.

## 7. Suggested follow-up audit sessions

1. **Safety-critical path audit (deletion correctness).** Adversarial review of `SafetyChecker` (661 ln), `DeletionGuard`, and every module's `clean()` against symlink races, TOCTOU, iCloud/Cloud-storage edge cases, and the recoverable-vs-permanent routing table in `CleanupFileRemover.swift`. Highest stakes in the codebase.
2. **Signing & notarization readiness.** What it takes to move from ad-hoc to Developer ID + notarized zips: entitlements review (sandbox stays off), hardened-runtime exceptions needed for `Process` shell-outs and Apple Events, release.yml changes, cask update.
3. **Features-layer decoupling review.** Inventory every `Process()`/network call inside `Sources/Features/`, define the target boundary (all side effects behind Core/Headless), and produce a migration list. Pairs with adding the first tests for `CLIExecutor` and scheduler.
4. **AI integration consolidation.** Single Anthropic client in Core, one model-selection config, keychain + TOML config unification, and a privacy note documenting exactly what leaves the machine per feature.
5. **Swift 6 strict-concurrency migration spike.** Turn on `-strict-concurrency=complete` per-target, measure the diagnostic count, and decide sequencing (Core first, Features last).
6. **CI hardening pass.** Pin Xcode + actions SHAs in ci.yml, add an `xcodegen`-regeneration drift check, wire `render-screenshots.sh` into CI as a smoke/visual gate, add changelog-vs-tag consistency check to `release.sh check`.
7. **Docs/agent-memory reconciliation.** Regenerate `ARCHITECTURE.md` from reality, decide tracked-vs-ignored policy for `.agents/TASKS`, backfill CHANGELOG 1.0.3/1.0.4/1.0.7.
